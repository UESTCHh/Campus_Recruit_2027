# 📘 TSN 核心流控机制：CBS (Credit-Based Shaper) 学习笔记

## 一、 CBS 核心概念与目标

### 1.1 CBS 的起源与背景

CBS（基于信用的整形器，Credit-Based Shaper）起源于 **IEEE 802.1Qav** 协议，该协议也被称为**时间敏感流的转发和排队协议（FQTSS，Forwarding and Queuing of Time-Sensitive Streams）**。这是在时间敏感网络（TSN，Time-Sensitive Networking）中解决带宽和时延挑战的核心机制。

**核心目标**：在普通异步数据流与时间敏感数据流共存的网络环境中，确保时间敏感流的服务质量（QoS）得到有效保障。

**通俗理解**：CBS 类似于"信用卡消费"机制。想发包必须有额度（信用值）；发包会扣额度；没包时不能提前攒额度。

### 1.2 应用场景

在 TSN 网络实际应用中，**AVB（音视频桥接）类消息流**（即流预留 SR 类）通常会采用 CBS 流控机制。

---

## 二、 核心组件与架构

IEEE 802.1Qav 协议包含**三个核心组件**：

### 2.1 优先级重映射

将不同优先级的消息流映射到不同的流量队列，建立起 **8 个不同消息流优先级与对应队列的关联**。

```
优先级队列映射表：
┌─────────────┬────────────┐
│  优先级 0-7  │  对应队列   │
├─────────────┼────────────┤
│  Priority 7 │  Queue 7   │  ← 最高优先级（时间敏感流）
│  Priority 6 │  Queue 6   │
│  Priority 5 │  Queue 5   │
│  Priority 4 │  Queue 4   │
│  Priority 3 │  Queue 3   │
│  Priority 2 │  Queue 2   │
│  Priority 1 │  Queue 1   │
│  Priority 0 │  Queue 0   │  ← 最低优先级（尽力而为流）
└─────────────┴────────────┘
```

### 2.2 流量整形（CBS 算法）

通过动态计算"信用值（Credit）"来控制发送权，确保时间敏感数据享有优先调度权。

### 2.3 队列管理

保证时间敏感流发生突发包时，网络仍能维持良好的性能和稳定性。

### 2.4 架构细节

具体在架构上，**每个配备了 CBS 的优先级都会有一个先进先出（FIFO）队列**，系统会为这个整形器提供和维护一个"信用值（Credit）"。

## 三、 四大核心参数（数学模型）

在 C++ 仿真中，每个启用了 CBS 的队列必须维护以下参数：

### 3.1 credit（传输信用值）

**单位**：bit

**定义**：它是决定队列能否发送数据的"货币"。

**发送条件**：
1. `credit ≥ 0`
2. 队列中有数据等待
3. 该队列的信用值在剩余队列中最大

**关于"剩余队列"的说明**：
- "剩余队列"**仅仅是指其他同样启用了CBS机制的队列**
- 非CBS队列（如普通数据流、TAS控制的高优先级流）没有信用值概念

```cpp
// 判断队列是否具备发送资格的伪代码
bool canTransmit(Queue& q) {
    return q.credit >= 0 &&           // 信用值非负
           !q.isEmpty() &&            // 队列有数据
           q.credit == getMaxCredit(); // 信用值最大（仅与其他CBS队列比较）
}
```

#### 3.1.1 CBS队列与非CBS队列的竞争机制

**1. 非CBS队列：没有"信用"概念**

在TSN/AVB网络中，除了音视频预留流（通常是SR Class A和Class B）会使用CBS之外，还有很多其他流量，它们**完全没有信用值（credit）的概念**，比如：

- **普通数据流（Best Effort, BE）**：比如网页浏览、后台下载数据。它们通常处于较低优先级，采用最基础的**严格优先级（Strict Priority, SP）**调度。有空就发，没空就等，不计算任何信用。

- **极高优先级的控制流（由TAS控制）**：比如车辆的紧急制动信号（紧急突发UB流或周期性控制流）。它们通过时间感知整形器（TAS）的"门控"机制（门开/门关）来绝对控制，也不看信用值。

**2. 只有"CBS圈子"内部才比拼信用**

信用值机制是专门为了约束**带有突发特性的时间敏感流**而设计的。通常交换机会拿出2个（或少数几个）优先级队列来开启CBS。

当提到"信用值比所有剩余队列信用值都大"时，它的应用场景是：
- 假设队列 A（视频流）和队列 B（音频流）都启用了 CBS
- 它们都在排队，且此时 credit 都 ≥ 0（都有资格发包）
- 此时系统会在 **CBS队列内部** 进行比较和调度，决定谁先占用端口

**3. CBS队列与非CBS队列如何竞争？**

整个端口调度的逻辑是分层的：

1. **先看门票**：CBS队列首先看自己的信用值是不是 ≥ 0。如果是负数，即使它的优先级再高，也被暂时"禁言"了，不能参与竞争。

2. **再拼等级**：如果CBS队列信用值 ≥ 0（拿到了发言门票），它就会带着自己的**优先级编号**（例如优先级5），去跟那些非CBS队列（比如优先级为3的普通BE队列）进行比拼。

3. **优先级高者得**：交换机发现优先级5的CBS队列有资格发包，且5 > 3，就会直接让CBS队列发包。在这个环节，是不看信用值大小的。

**总结**：信用值只是CBS机制对自己名下流量的一种"自我约束"和"内部分配"工具。其他不归CBS管的队列，根本不在乎也不计算信用值。

### 3.2 portTransmitRate（端口物理传输速率）

**单位**：bit/s

**定义**：端口物理传输速率，通常是固定的（如 1Gbps、10Gbps）。

### 3.3 idleSlope（空闲斜率）

**单位**：bit/s

**定义**：信用值的"累积速率"。

**触发条件**：当至少有一个数据帧在队列中等待传输（但当前无法传输）时，credit 以这个速度增加。

**通俗理解**：你在排队等候时，信用值在不断积累。

### 3.4 sendSlope（发送斜率）

**单位**：bit/s

**定义**：信用值的"消耗速率"。

**计算公式**：
```
sendSlope = idleSlope - portTransmitRate
```

**重要特性**：由于物理端口速率必然大于分配带宽，因此 `sendSlope` 必然是一个**负数**。

### 3.5 hiCredit / loCredit（信用上下限）

**hiCredit**：信用上限，防止网络极度空闲时无限积攒额度，导致后续产生超级突发。

**loCredit**：透支下限，防止发送超大数据包后透支过深，导致永远无法"还清债务"。

```
信用值变化范围示意图：
        hiCredit
          ↑
          │    idleSlope ↑
          │              │
  Credit  │              │
          │    sendSlope ↓（负值）
          │              │
          ↓              │
       loCredit          │
          （通常为负值）  │
                         │
                      Credit = 0（信用基准线）
```

### 3.6 maxFrameSize / maxInterferenceSize

| 参数 | 定义 |
|------|------|
| maxFrameSize | 队列允许传输的最大帧长度 |
| maxInterferenceSize | 网络中可能存在的最大干扰帧长度 |

这些参数主要用于计算 `hiCredit` 和 `loCredit`：
```
hiCredit = idleSlope × (maxFrameSize / portTransmitRate)
loCredit = sendSlope × (maxFrameSize / portTransmitRate)
```

## 四、 CBS 核心状态机（仿真运转逻辑）

### 4.1 三种状态流转

在每个时间步进中，CBS 队列会在以下**三种状态**中流转：

#### 状态 1：空闲 (Idle)

- **条件**：队列为空（没有数据包要发）。
- **动作**：`Credit` 强制重置为 **0**。这是绝对铁律，严禁队列在空闲时"提前攒钱"。

```cpp
void updateIdle(Queue& q) {
    if (q.isEmpty()) {
        q.credit = 0; // 强制清零，禁止攒信用
    }
}
```

#### 状态 2：排队积攒 (Waiting)

- **条件**：队列有包，但端口被占用，或者当前 `Credit < 0`（处于透支惩罚期）。
- **动作**：`Credit` 以 `IdleSlope` 的速率随时间线性增加（最高不超过 `hiCredit`）。

```cpp
void updateWaiting(Queue& q, double deltaTime) {
    if (!q.isEmpty() && q.isPortBusy()) {
        // 等待期间，信用值累积
        q.credit = min(q.credit + q.idleSlope * deltaTime, q.hiCredit);
    }
}
```

#### 状态 3：发送 (Transmitting)

- **条件**：队列有包，具备发送资格，且端口空闲分配给了该队列。
- **动作**：在发包持续期间，`Credit` 以 `SendSlope` 的速率线性下降（极大概率会降为负数）。

```cpp
void updateTransmitting(Queue& q, double deltaTime) {
    if (q.isTransmitting()) {
        // 发送期间，信用值消耗（sendSlope 为负值）
        q.credit += q.sendSlope * deltaTime;
    }
}
```

### 4.2 状态转换图

```
                    ┌─────────────────┐
                    │                 │
                    │   Idle (空闲)    │◄──────────────────┐
                    │   credit = 0     │                   │
                    │                 │                   │
                    └────────┬────────┘                   │
                             │                             │
                    队列有数据包                        队列无数据包
                             │                             │
                             ▼                             │
                    ┌─────────────────┐                   │
                    │                 │                   │
                    │   Waiting (积攒)  │───────────────────┘
                    │   credit +=      │
                    │   idleSlope      │
                    │                 │
                    └────────┬────────┘
                             │
                    端口空闲 + credit >= 0
                             │
                             ▼
                    ┌─────────────────┐
                    │                 │

                    │ (发送)           │
                    │ credit +=        │

                    发送完成 + 队列非空

                    │                 │

                    │   (继续积攒)     │
                    │                 │
                    └─────────────────┘
```

## 五、 CBS 的工作原理解析

### 5.1 本质理解

> **CBS 的本质是"有信用才能发包，发包就扣信用，等待就涨信用"**

### 5.2 场景 1：无冲突帧情况

当数据帧排队时，如果输出端口空闲，数据帧立即出队传输。

**执行过程**：
```
时间线：0 ───────────────────────────────►

1. 数据帧到达队列，credit = 0
   ┌────┐
   │数据包│ → 立即发送
   └────┘

2. 传输期间，credit 以 sendSlope 速率下降（变为负值）
   credit: 0 → -1000 → -2000 → -3000 → ...

3. 传输结束后，若还有数据等待，credit 以 idleSlope 速率回升
   credit: -3000 → -2000 → -1000 → 0

4. 只有当 credit >= 0 时，才能发送下一帧
```

### 5.3 场景 2：存在冲突帧情况

如果数据准备发送时，输出端口正处于繁忙状态（被其他冲突帧占用），数据帧只能等待。

**执行过程**：
```
时间线：0 ───────────────────────────────►

1. 数据帧到达队列，端口被占用，只能等待
   ┌────┐  等待中...
   │数据包│
   └────┘

2. 等待期间，credit 以 idleSlope 速率增加
   credit: 0 → +500 → +1000 → +1500 → ...

3. 冲突结束，帧开始传输
   credit: +1500 → +1000 → +500 → 0 → -500 → ...

4. credit 降为负值，但仍需继续发送（已获得发送权）
```

### 5.3.1 冲突帧的详细解释

**冲突帧的来源**：冲突帧**正是来自于其他优先级的队列**，而不是自己这个队列。

#### 1. 同队列内部：是"排队"而不是"冲突"

在您所关注的这个 CBS 队列（例如优先级为 A 的队列）内部，数据帧是严格按照**先进先出（FIFO）**的规则排队的：
- 队头的帧负责在最前面等待物理端口的发送权
- 排在后面的帧只是在排队等待前面的帧发完

因此，同一个队列内部的帧不存在去"抢占"物理端口的情况，它们井然有序。

#### 2. 物理端口的唯一性：多队列竞争

一个交换机的物理输出端口在**同一微秒/纳秒时刻，只能发送一个数据帧**。

但是，这个端口背后通常绑定了多个不同优先级的队列（在 IEEE 802.1Q 标准中最多有 8 个队列）。除了当前的 CBS 流队列，还可能存在：
- **更高优先级的队列**：比如由 TAS（时间感知整形）控制的、有着严格时间窗口的控制周期流
- **更低优先级的队列**：比如最普通的 Best Effort（尽力而为）后台下载流量

#### 3. 产生冲突的真实场景

当您的 CBS 队列中积累了数据，且信用值（credit）≥ 0，具备了发送资格时，如果物理端口"正忙"，通常是以下两种情况造成的：

**情况 1：低优先级流的"占位"**
- 虽然别的队列优先级比 CBS 低，但如果 CBS 数据帧刚好到达时，那个低优先级队列的数据帧**恰好已经开始发送了**
- 由于以太网帧默认是不可打断的（除非开启了 802.1Qbu 帧抢占机制），CBS 队列就必须被迫等待这个低优先级大包发完
- 这就是经典的"头端阻塞"（Head-of-Line Blocking）

**情况 2：高优先级流的"插队"**
- 端口刚好到了高优先级（如 TAS 流）的专属发送时间窗口
- 高优先级队列的数据帧拿到了发送权，CBS 队列也必须等待

#### 4. 信用值补偿机制的意义

正是因为端口会被**其他优先级队列**的流量占用，CBS 机制才设计了**"等待期间增加信用值（idleSlope）"**的补偿逻辑。

这个逻辑确保了：即使 CBS 流被别人（其他队列）占用了时间，它也会在等待中不断攒够"人品（信用值）"。一旦那个霸占端口的帧发送完毕，CBS 队列就会凭借最高积累的信用值，强势夺回物理端口的发送权！

### 5.4 防突发机制（关键设计）

> **如果传输结束后，credit 仍然大于 0，但队列中已经没有等待发送的数据了，系统会直接将 credit 清零。**

这一设计非常关键，它防止了空闲队列无限积攒信用，从而避免了未来突然产生大量突发流量冲垮网络。

```cpp
void onTransmissionComplete(Queue& q) {
    if (q.isEmpty() && q.credit > 0) {
        // 队列已空但信用值仍为正，立即清零
        q.credit = 0; // 防止攒出"超级突发"
    }
}
```

## 六、 传统以太网 vs CBS 网络

### 6.1 传统以太网 (Strict Priority, 严格优先级) 的问题

#### 问题一：低优先级"被饿死"

- 高优先级流只要有包就一直霸占端口
- 低优先级数据只能无期限死等
- **现象**：某些关键控制报文可能永远无法发送

#### 问题二：恐怖的流量突发

- 高优先级数据包首尾相连，形成密集的"车队"冲向下游交换机
- 瞬间打满缓存
- 造成极大抖动和丢包

### 6.2 CBS 的解决之道："强制深呼吸"

1. **CBS 发包会扣除信用**。一旦信用变为负数，即使是最高优先级的流也必须**强制暂停（深呼吸）**，等待信用恢复。

2. 这个短暂的暂停间隙，让交换机有机会去发送低优先级的包，解决了"饿死"问题。

3. 原本密集的突发流量被均匀地拉开间距，变成了平缓的溪流，极大地提升了全网的稳定性。

### 6.3 对比总结

| 特性 | 严格优先级 | CBS |
|------|----------|-----|
| 带宽保障 | 无 | 有 |
| 突发抑制 | 无 | 有 |
| 低优先级饿死 | 可能 | 不会 |
| 延迟确定性 | 差 | 好 |
| 实现复杂度 | 低 | 中 |

## 七、 CBS 与 TAS 联合调度

### 7.1 TAS（时间感知整形器）简介

TAS（Time-Aware Shaper，时间感知整形器）是 IEEE 802.1Qbv 定义的流控机制，用于控制有严格确定性周期的控制流。

### 7.2 联合调度策略

在更广泛的 TSN 网络工程中，CBS 通常需要与 TAS 等其他流控机制联合调度：

| 机制 | 适用场景 | 特点 |
|------|---------|------|
| **TAS** (IEEE 802.1Qbv) | 严格确定性周期的控制流 | 周期性调度，时间窗口精确控制 |
| **CBS** (IEEE 802.1Qav) | 带有突发特征的预留音视频流 | 信用控制，带宽保障 |

**两者结合**：TAS 控制有严格时间要求的控制流，CBS 约束和保障突发性的 AVB 流，共同提升网络的实时性与可靠性。

## 八、 🚨 核心开发避坑：标准协议 vs 项目定制

### 8.1 IEEE 官方标准（不要用）

`Credit ≥ 0` 仅仅是"资格审查"，最终发包权依然严格由静态优先级（Priority）决定。**Credit 的绝对值大小没有任何比较意义**。

### 8.2 《项目建议书》定制算法（必须遵循）

你的项目采用了一种基于信用的改良竞争算法。文档明确要求：

> **比较不同优先级队列的信用值，并选择具有"最大 Credit 值"的队列进行数据发送。**

### 8.3 C++ 编码准则

```cpp
/**
 * @brief 选择下一个发送队列
 * @return 具有最大信用值的非空队列索引，若无合格队列返回 -1
 */
int selectNextQueue(vector<Queue>& queues) {
    int selectedIndex = -1;
    int maxCredit = -1;

    for (int i = 0; i < queues.size(); ++i) {
        Queue& q = queues[i];

        // 检查队列是否有数据且信用值足够
        if (!q.isEmpty() && q.credit >= 0) {
            // 选择信用值最大的队列
            if (q.credit > maxCredit) {
                maxCredit = q.credit;
                selectedIndex = i;
            }
        }
    }

    return selectedIndex;
}
```

## 九、 知识点补充与个人理解

### 9.1 为什么需要信用值上限 (hiCredit)？

如果没有 hiCredit 限制，当网络长时间空闲时：
1. 队列会持续积攒信用
2. 信用值可能达到一个巨大的正数
3. 当突发流量到来时，这个队列可以一次性发送远超平均带宽的数据量
4. 这会导致下游交换机缓存被打满，引发丢包和抖动

**解决方案**：设置 hiCredit，限制最大积攒信用，防止突发流量冲击。

### 9.2 为什么需要信用值下限 (loCredit)？

如果没有 loCredit 限制，当发送一个超大帧时：
1. credit 会急剧下降到一个巨大的负数
2. 队列需要很长时间才能恢复信用（信用变为正数）
3. 在恢复期间，即使有紧急数据也无法发送
4. 这会导致关键消息的延迟不可接受

**解决方案**：设置 loCredit（通常为负值），限制最大透支深度，保证信用恢复时间可控。

### 9.3 sendSlope 为什么是负值？

```
sendSlope = idleSlope - portTransmitRate

例如：
- portTransmitRate = 1000 Mbps (端口速率)
- idleSlope = 100 Mbps (预留带宽)
- sendSlope = 100 - 1000 = -900 Mbps
```

**含义**：每发送 1 秒数据，信用值消耗 900 Mbit（因为其中只有 100 Mbit 是"自己的"，另外 900 Mbit 是"借来的"）。

### 9.4 CBS 与令牌桶算法的区别

| 特性 | CBS | 令牌桶 |
|------|-----|--------|
| 控制方式 | 信用值累积/消耗 | 令牌累积/消耗 |
| 突发容忍 | 低（有 hiCredit 限制） | 高（令牌可累积） |
| 延迟特性 | 更稳定 | 可能突发 |
| 应用场景 | TSN 音视频流 | 普通 QoS |

### 9.5 实际配置建议

在 TSN 网络工程中，配置 CBS 参数时需要考虑：

1. **idleSlope 设置**：通常等于该队列预留的带宽
2. **maxFrameSize**：根据实际应用的最大帧长设置（如 1518 bytes）
3. **hiCredit 计算**：
   ```
   hiCredit = idleSlope × (maxInterferenceSize / portTransmitRate)
   ```
4. **loCredit 计算**：
   ```
   loCredit = sendSlope × (maxFrameSize / portTransmitRate)
   ```

### 9.6 个人感悟

CBS 机制的核心思想是**"削峰填谷"**：

- **削峰**：通过信用值上限，防止突发流量冲击网络
- **填谷**：通过信用值累积，让低优先级流有机会发送

这种设计体现了**"公平共享带宽"**的理念：每个队列只能使用自己"挣来"的信用值，不能无限透支，从而保证了网络的稳定性和可预测性。

在实际项目中，理解 CBS 的状态机流转是实现的核心。只有准确判断队列所处的状态（Idle/Waiting/Transmitting），并正确更新信用值，才能保证仿真结果的准确性。
