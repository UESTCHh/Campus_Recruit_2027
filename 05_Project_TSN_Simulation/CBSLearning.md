# 📘 TSN 核心流控机制：CBS (Credit-Based Shaper) 学习笔记

## 一、 CBS 核心概念与目标
CBS（基于信用的整形器）是 IEEE 802.1Qav 标准定义的核心机制，主要用于音视频流（AVB 流）等需要带宽保障的传输。
* **核心目标**：平滑流量（削峰填谷）、防止网络突发（Burst）、保障预留带宽。
* **通俗理解**：类似“信用卡消费”。想发包必须有额度（信用值）；发包会扣额度；没包时不能提前攒额度。

## 二、 四大核心参数（数学模型）
在 C++ 仿真中，每个启用了 CBS 的队列必须维护以下参数：

1. **当前信用值 ($Credit$)**：动态变化的余额，决定了当前队列是否具备发包资格（通常要求 $Credit \ge 0$）。
2. **空闲斜率 ($IdleSlope$)**：积攒信用的速率，通常等于为该队列预留的分配带宽。
3. **发送斜率 ($SendSlope$)**：消耗信用的速率。
   * **公式**：$SendSlope = IdleSlope - PortTransmitRate$（端口总物理速率）。
   * **注意**：由于物理端口速率必然大于分配带宽，因此 $SendSlope$ 必然是一个**负数**。
4. **信用上下限 ($hiCredit$ 与 $loCredit$)**：
   * $hiCredit$：信用上限，防止网络极度空闲时无限积攒额度，导致后续产生超级突发。
   * $loCredit$：透支下限，防止发送超大数据包后透支过深，导致永远无法“还清债务”。

## 三、 CBS 核心状态机（仿真运转逻辑）
在每个时间步进中，CBS 队列会在以下三种状态中流转：

* **状态 1：空闲 (Idle)**
  * **条件**：队列为空（没有数据包要发）。
  * **动作**：$Credit$ 强制重置为 **0**。这是绝对铁律，严禁队列在空闲时“提前攒钱”。
* **状态 2：排队积攒 (Waiting)**
  * **条件**：队列有包，但端口被占用，或者当前 $Credit < 0$（处于透支惩罚期）。
  * **动作**：$Credit$ 以 $IdleSlope$ 的速率随时间线性增加（最高不超过 $hiCredit$）。
* **状态 3：发送 (Transmitting)**
  * **条件**：队列有包，具备发送资格，且端口空闲分配给了该队列。
  * **动作**：在发包持续期间，$Credit$ 以 $SendSlope$ 的速率线性下降（极大概率会降为负数）。

## 四、 传统以太网 vs CBS 网络

### 1. 传统以太网 (Strict Priority, 严格优先级) 的两宗罪
* **低优先级“被饿死”**：高优先级流只要有包就一直霸占端口，低优先级数据只能无期限死等。
* **恐怖的流量突发**：高优先级数据包首尾相连，形成密集的“车队”冲向下游交换机，瞬间打满缓存，造成极大抖动和丢包。

### 2. CBS 的解决之道：“强制深呼吸”
* CBS 发包会扣除信用。一旦信用变为负数，即使是最高优先级的流也必须**强制暂停（深呼吸）**，等待信用恢复。
* 这个短暂的暂停间隙，让交换机有机会去发送低优先级的包，解决了“饿死”问题。
* 原本密集的突发流量被均匀地拉开间距，变成了平缓的溪流，极大地提升了全网的稳定性。

## 五、 🚨 核心开发避坑：标准协议 vs 项目定制
在编写“端口空闲时选哪个队列发包”的 C++ 仲裁逻辑时，必须注意：

* **IEEE 官方标准（不要用）**：$Credit \ge 0$ 仅仅是“资格审查”，最终发包权依然严格由静态优先级（Priority）决定。$Credit$ 的绝对值大小没有任何比较意义。
* **《项目建议书》定制算法（必须遵循）**：你的项目采用了一种基于信用的改良竞争算法。文档明确要求：**比较不同优先级队列的信用值，并选择具有“最大 Credit 值”的队列进行数据发送。**
* **你的 C++ 编码准则**：在遍历队列寻找下一个发送包时，不仅要看有没有包、$Credit$ 是否 $\ge 0$，还要编写逻辑**找出当前所有队列中 $Credit$ 值最大的那一个**，赋予其端口发送权。
