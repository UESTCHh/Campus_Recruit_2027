## 一、 操作符重载与临时对象

在设计复数的加法 `operator+` 时，我们会遇到一个极其经典的面试题。

### 💡 面试核心考点：为什么 `operator+` 不能返回引用？
* 因为两个复数相加（例如 `c1 + c2`），必定会产生一个**全新的复数**。这个新的复数是在函数内部刚刚创建出来的**局部变量 (Local object)**。如果返回它的引用，函数一结束，这个局部变量就死亡销毁了。外界拿到的是一个指向已死亡内存的悬空引用（野指针）！
* **结论：** 凡是函数内部新创建的局部对象，**必须按值返回 (Return by value)**，绝对不能返回引用。

### 📌 语法糖：临时对象 (Temporary Object)
为了极致的效率，我们连局部变量的名字都不取，直接用 `类名(参数)` 创建一个没有名字的临时对象返回。它的生命周期极短，只在这一行代码有效。
```cpp
inline Complex operator+(const Complex& x, const Complex& y) {
    // Complex(...) 就是一个临时对象，连名字都没有，直接作为值返回
    return Complex(x.real() + y.real(), x.imag() + y.imag());
}
```

## 二、 为什么要有“三大件”？(The Big Three)
如果一个类里面有指针（例如 String 类里有 char* m_data），我们就绝对不能使用编译器默认生成的拷贝函数，必须自己写以下三个函数（合称三大件）：
* 1.析构函数 (Destructor)

* 2.拷贝构造函数 (Copy Constructor)

* 3.拷贝赋值函数 (Copy Assignment Operator)

## 三、 致命的内存灾难：深拷贝 vs 浅拷贝
### 💡 面试核心考点：如果带指针的类用了编译器默认的拷贝（浅拷贝），会发生什么？
默认的拷贝是 **“浅拷贝”（只拷贝指针的值，也就是内存地址）。**
假设字符串 A 包含了指针指向 "Hello"，用 A 去拷贝构造 B。浅拷贝会导致 A 和 B 的指针指向同一块内存空间。
**引发两大灾难：**
* 1.Double Free（重复释放）： 当 A 和 B 的生命周期结束调用析构函数时，这块内存会被 delete 两次，程序直接崩溃！

* 2.Memory Leak（内存泄漏）： 如果 B 原本自己有一块内存空间，浅拷贝直接把 B 的指针改向了 A 的内存，B 原本的内存就再也找不到了，发生泄漏。
### 🛠️ 正确做法：深拷贝
自己写拷贝构造函数，必须先 new 一块一样大的新内存，然后再用 strcpy 把内容真正地复制过去。
```CPP
// String 类的拷贝构造函数（深拷贝）
inline String::String(const String& str) {
    m_data = new char[strlen(str.m_data) + 1]; // 1. 分配新空间
    strcpy(m_data, str.m_data);                // 2. 复制真实内容
}
```

## 四、 拷贝赋值的救命操作：自我赋值检查
在重载 = 号（拷贝赋值函数）时，第一步必须检查是不是“自己赋值给自己”。

```CPP
inline String& String::operator=(const String& str) {
    // 救命的防线：自我赋值检查 (Self Assignment Check)
    if (this == &str) {
        return *this; 
    }
    
    delete[] m_data; // 1. 杀掉原本的自己
    m_data = new char[strlen(str.m_data) + 1]; // 2. 重新分配大小
    strcpy(m_data, str.m_data); // 3. 复制内容
    
    return *this;
}
```
---

### 💡 面试核心考点：如果不加 if(this == &str) 会引发什么后果？
如果是自己赋值给自己（例如 a = a;），代码第一步就是 delete[] m_data;，把自己的内存清空了。等到第三步去拿源数据复制时，源数据已经变成了乱码/野指针！不写这行检查，面试直接挂。

## 五、 数组的毁灭：new[] 必须配 delete[]

### 💡 面试核心考点：用 delete 释放数组，和用 delete[] 有什么区别？
当你写出 String* p = new String[3]; 时，内存里创建了 3 个 String 对象。

* 如果你用 delete[] p;：编译器知道这是一个数组，会调用 3 次析构函数，把 3 个对象内部的 char* 内存全部释放，完美！

* 如果你只用 delete p;：编译器以为这只是一个对象，只调用 1 次析构函数。剩下的 2 个对象本身虽然被销毁了，但它们内部 m_data 指向的那片动态内存永远不会被释放，引发严重内存泄漏！

## 六、 内存的宿命：栈 (Stack) 与 堆 (Heap)
面试官经常会指着一段代码问：“这个对象分配在哪里？什么时候销毁？”
在 C++ 中，内存分配最核心的就是搞懂栈和堆的区别。

### 1. 栈 (Stack)：自动售货机（作用域内的局部变量）
如果你在大括号 `{}`（也就是一个作用域，比如函数体内部）里写下：
```cpp
{
    Complex c1(1, 2); // c1 分配在栈上
} // 离开大括号的瞬间，c1 自动死亡（调用析构函数）
```
* 宿命： 栈内存由编译器完全接管。只要程序运行离开了这个大括号 {}，栈上的对象就会被自动回收。

* 特点： 极度安全、速度极快，但空间有限。你不需要（也不能）去 delete 它。

### 2. 堆 (Heap)：租车行（用 new 动态分配的内存）
如果你使用了 new 关键字：

```cpp

{
    Complex* p = new Complex(1, 2); // p 这个指针本身在栈上，但它指向的真实数据在堆上
} // 离开大括号，指针 p 死了，但堆里的数据依然活着！
```
* 宿命： 堆内存是你向操作系统“借”来的。你就是这块内存的绝对主宰。只要你不写 delete p;，这块内存即使到了世界末日（程序结束前）也一直存在！

* 致命风险（Memory Leak）： 在上面的例子中，离开大括号后，指针 p 没了，你再也找不到那块堆内存的地址了，但它依然占据着内存空间，这就是极其可怕的内存泄漏。

### 💡 终极铁律：
凡是用 new 分配在堆上的对象，不论程序跑到哪里，都必须由你自己亲手写下 delete 来斩断它的宿命！
