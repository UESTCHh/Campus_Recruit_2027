## 一、 类的补充特性：static 与 Template以及单例模式

### 1. 独立于对象的 `static`
普通的成员变量，每个对象都有一份；而加上 `static` 后，这个变量就脱离了具体的对象，**全班只有一份，大家共享**。
* **面试核心考点：** `static` 成员函数和普通成员函数的根本区别是什么？
  **答案：** 普通成员函数在底层偷偷带着一个 `this` 指针（代表当前调用它的对象）。而 `static` 成员函数**没有 `this` 指针**！所以它只能处理 `static` 的成员变量，绝对不能调用普通的成员变量。

### 2. 泛型编程的基石：`template` (模板)
当你发现除了数据类型不同（比如一个是 `int`，一个是 `double`），其他逻辑完全一样时，就把类型抽出来变成模板 `template <typename T>`。让编译器在编译时帮你“自动生成”代码。

### 3. 🔥 大厂必考手撕代码：单例模式 (Singleton)
既然 `static` 只有一份，侯捷老师引出了极其经典的“单例模式”（保证一个类永远只能创建一个对象）。
面试时如果让你写单例，直接写下面这种 **Meyers Singleton（局部静态变量懒汉式）**，不仅代码极简，而且在 C++11 之后是绝对线程安全的！

```cpp
class Singleton {
public:
    // 3. 提供一个全局访问点，返回唯一实例的引用
    static Singleton& getInstance() {
        static Singleton instance; // 静态局部变量：只有在第一次调用时才会创建，且只创建一次！
        return instance;
    }

private:
    // 1. 必须把构造函数私有化，禁止外界 new 对象
    Singleton() {}
    
    // 2. 最好也把拷贝构造和拷贝赋值删掉（C++11 写法），防止被意外复制
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;
};

// 外界调用方式：
// Singleton::getInstance().someFunction();
```

## 二、 类与类的三大关系

大厂系统架构设计，本质上就是玩转这三种关系。

### 1. Composition (复合, has-a)
* **含义：** 一个类里面**直接包含**了另一个类的对象。比如标准库的 `queue` 里面包含了一个 `deque`。这就是 `has-a`（我有一个你）。
* **生死羁绊：** 同生共死。
  * **构造由内而外：** 先调用内部组件的默认构造函数，再执行自己的构造函数。（盖楼先打地基）
  * **析构由外而内：** 先执行自己的析构函数，再调用内部组件的析构函数。（拆楼先拆房顶）

### 2. Delegation (委托, has-a pointer)
* **含义：** 一个类里面包含了另一个类的**指针**（比如著名的 Pimpl 惯用法，Pointer to Implementation）。
* **核心优势（极度解耦）：** 因为只是一个指针，内部组件的具体实现哪怕改得翻天覆地，外部类的客户端代码也完全不需要重新编译。也就是侯捷老师说的 **“编译防火墙”**。它的生命周期不一定同步，可以在需要的时候再 `new` 出来。

### 3. Inheritance (继承, is-a)
* **含义：** 子类继承父类的一切数据和函数。比如“麻雀 `is-a` 鸟”。
* **生死羁绊：**
  * **构造由内而外：** 先调用父类（Base）的默认构造函数，再执行子类（Derived）的构造。
  * **析构由外而内：** 先执行子类（Derived）的析构，再调用父类（Base）的析构。
  
#### 💡 面试终极追问：如果一个子类继承了父类，同时它自己又包含了一个组件类（复合），构造顺序是什么？
* 答案： 先调用父类的构造函数，再调用组件类的构造函数，最后执行自己的构造函数。

## 三、 虚函数与经典设计模式

在继承关系中，成员函数的三种设计哲学：
1. **非虚函数 (Non-virtual)：** `void func();` 
   你不希望子类去重新定义（Override）它。父类的规矩，子类照做就行。
2. **纯虚函数 (Pure Virtual)：** `virtual void func() = 0;`
   父类完全不知道怎么实现，**强制子类必须去实现**。带有纯虚函数的类叫“抽象类”。
3. **虚函数 (Virtual)：** `virtual void func();`
   父类提供了一个默认的实现，但**允许且鼓励子类去重新定义它**。

### 🎯 经典应用一：Template Method (模板方法模式) -> 继承 + 虚函数
侯捷老师举了微软 MFC 框架 `CWinApp` 的例子，这是大厂框架开发的绝对核心思路：
1. 父类（框架）写好了一个极其复杂的通用流程（比如 `OnFileOpen` 函数）。
2. 在这个复杂流程中，有一步（比如 `Serialize` 读写文件）是无法确定的，必须由未来的应用程序来决定。
3. 于是，父类把这一步设为**虚函数**。
4. 子类继承父类，**重写（Override）** 这个虚函数。
5. 运行结果：子类的对象调用 `OnFileOpen` 时，执行到一半，通过 `this` 指针（实际上指向子类对象）的引导，**神奇地跑去调用了子类重写的 `Serialize`**！

### 🎯 经典应用二：Observer (观察者模式) -> 委托 + 继承
在第 13 集中，侯捷老师展示了如何把“委托”和“继承”结合起来。当你做了一个文件的数据（Subject），有很多窗口（Observer，如饼图、柱状图）都需要看着这个数据，数据一变，窗口全跟着变。

* **做法：** 数据类里面放一个 vector<Observer*>（这是**委托**，拥有指针集合）。

* 当数据发生改变时，只需要遍历这个 vector，挨个调用 obs->update() 即可。未来你想增加新的图表类型，只需要新建一个类**继承**自 Observer 并重写 update 函数，塞进 vector 里，原有的数据类代码一行都不用改！

### **💡 面试一句话总结：**
所谓的多态初步体验，就是“父类的指针或引用，指向了子类的对象，从而在运行期动态地调用了子类重写后的虚函数”。
